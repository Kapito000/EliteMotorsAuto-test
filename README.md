# Решение.
Для работы ключевой механики в классе `Movement` выполняется запуск луча в центр притяжения. Луч сталкивается с поверхностью куба. У столкновения получаем вектор нормали, который будет указывать локально направление "вверх".

Все движение игрока просчитывается относительно его локальных осей координат. В том числе и притяжение на игрока действует в направлении "голова-ноги" вдоль локально оси `y`.

Каждый кадр игрок стремится повернуться локально осью `y`, таким образом, чтобы она была параллельна текущей нормали поверхности.

Таким образом происходит изменение направления силы притяжения.

## Ввод.
`IInputService` дает возможность подписаться на управляющие события, а также выполнить симулирование управления. В данном случаем симуляцией пользуются кнопки UI. В "боевом" проекте имеет смысл отделить от `IInputService`интерфейс, позволяющий вызывать симуляцию управления.

В качестве схемы ввода использую стандартную `InputSystem_Actions`, идущую "из коробки" при создании нового проекта. Ее функционала вполне достаточно.

# Архитектура.
Чтобы не переусложнить решение - выделил статичный класс `G` (от GlobalContext). С его помощью получаю зависимость на сервис ввода `IInputService`.

В "боевом" проекте лучше использовать DI (Zenject, VContainer) или сервис локатор. Зависит от требований.
